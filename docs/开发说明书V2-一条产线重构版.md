# 一条视频的自动化产线 —— 开发说明书 V2（重构蓝本）

> 版本：**v2.0（重构规划稿，用于从零重做本项目）**
> 日期：2026-07-11
> 前身：`开发说明书.md`（v1.0，12 条平台流水线的 as-built 文档）。V1 的技术选型与踩坑记录大量继承，产品模型推倒重来。
> 运行形态：个人电脑本地应用（本机 Node 服务 + 浏览器界面；后期可选 Electron 桌面壳）

---

## 0. 为什么重做（V1 的教训）

V1 把"平台差异"建模在了错误的层级：

1. **平台差异是参数级的，却被做成了流程级的。** 抖音视频与小红书视频的流水线步骤结构 100% 相同，Prompt 模板 90% 逐句相同，真正的差异只有封面比例、标题字数上限、几句话术。结果是 12 个流水线 JSON + 48 个 Prompt 模板互相复制，改一处通用逻辑要改 N 处。
2. **语义反了。** 勾选 4 个平台 = AI 独立生成 4 遍标题/口播稿/分镜 = 4 条内容不同的视频、4 倍成本。而实际需求是"做好**一条**视频，往多个平台分发"。
3. **宽度过剩、深度不足。** 12 条流水线铺得很宽，但最核心的"从选题到一条能直接上传的 mp4"没有打穿——视频链路停在剪映草稿 + 分镜 CSV，成片仍要人进剪映手工操作。

**V2 的一句话定位：一条视频的自动化产线 + 一张平台参数表。** 选题进去，一条成片 mp4 出来，末端按参数表派生成 N 个平台的发布包。

## 1. 产品模型

### 1.1 核心公式

```
V1：选题 × N 个平台流水线 = N 次独立生成（内容不同、成本 N 倍）
V2：选题 × 1 条产线 = 1 条母版成片 → 派生 × N 个平台发布包（内容同源、派生近乎零成本）
```

### 1.2 主线（唯一的一条流水线）

```
选题录入(三种信息源) → 素材理解 → 标题(5候选) → 口播稿 → 分镜表 → 逐镜头声画物料 → TTS配音 → ffmpeg合成母版mp4 → 评审 → 平台派生 → 发布包ZIP
    人工             AI看图/看视频  AI,可卡点     AI      AI(JSON)    出图/实拍切片      免费/API    ★本次重构的核心      AI+规则   参数表驱动    人工发布
```

- **母版（master）**：9:16 竖版 1080×1920 成片。抖音 / 小红书 / 视频号 / B 站竖屏可直接用；其他比例在派生步骤转换。
- **派生（adapt）**：不重新创作。同一条母版转比例 + 按平台参数小幅改写标题文案 + 封面派生尺寸 + 生成发布注意事项清单，打包成"发布包"。
- V1 的图文/长文/图集/MV 流程**全部不做**（见 §10 非目标）。将来若要图文，作为独立产线另立项，不塞回本项目。

### 1.3 三种信息源，一条产线

视频的原始信息来自三种入口。**信息源只改变产线的"入料"，不分叉产线本身**——三者在分镜表处汇合（每个镜头用 `source` 字段标记画面来源，见 §5.3），之后的 TTS / 合成 / 评审 / 派生完全共用：

| 模式 | 输入 | 画面从哪来 | 典型场景 |
|---|---|---|---|
| **A1 纯文本** | 主题 + 描述/梗概 | 全部 AI 出图（可选图生视频） | 知识 / 观点 / 清单类内容 |
| **A2 参考图** | 一张有内容的图片 + 提示词 | AI 出图，以参考图为内容与风格锚点（图生图 / 提示词注入） | 围绕一张好图展开叙事 |
| **B 实拍素材** | 手机随手拍的 10~30 秒视频 + 一两句简述 | **实拍片段按时间戳切片**为主，AI 只补少量生成镜头（片头钩子 / 片尾收尾） | 生活记录 + AI 旁白解说 |

- **B 模式是扩宽度的第一优先级**：没有出图的不确定性，AI 只负责"看懂素材 → 写旁白 → 配音字幕 BGM 成片"，实现成本最低、成片效果最稳。
- A2 的参考图同时服务三处：素材理解（vision 读图注入选题上下文）、出图锚点（图生图 / 参考图参数）、封面底图候选。
- 深度优先原则不变：M1~M3 先用 A1 打穿成片，M4 再接入 A2 / B（见 §11）。

### 1.4 运行形态（回答"HTML 里能不能调 API"）

**不能用纯 HTML 静态页面**，原因有三：① 主流 AI API 禁止浏览器跨域直连（CORS）；② key 写进前端等于泄露；③ 浏览器无法运行 ffmpeg / TTS / 读写本地文件，而成片链路全靠它们。

**正确形态（与 V1 相同，直接继承）**：本机 Node 服务（127.0.0.1:8787）承担所有 API 调用、key 保管、ffmpeg 执行、文件读写；浏览器只是操作界面。`pnpm dev` 起服务后浏览器打开即用——这已经是完整的本地软件。若要"双击图标启动"的桌面形态，最后阶段用 Electron 把同一个服务包一层壳（V1 已验证可行），**不作为前期目标**。

```
├──────────────────────┼─────────────────────────────────┤
│  引擎适配层 packages/providers（继承 V1，砍掉 web 适配器）    │

│ compose   │ adapt(platforms.json) │ review(继承V1词库)      │
│ (ffmpeg)  │ (转比例/改写/打包)      │                        │
├──────────┴──────────────────────┴──────────────────────┤
│  存储：SQLite（node:sqlite）+ 文件系统 workspace/           │
└────────────────────────────────────────────────────────┘
```

与 V1 的结构差异：

| | V1 | V2 |
|---|---|---|
| 流水线 | 12 条 JSON + 通用 DAG 编排引擎 | **1 条固定主线**，代码内写死顺序（分镜画面/TTS 逐镜头并行），不再需要通用 DAG |
| Prompt 模板 | 48 个（按平台×模式×步骤） | **6~8 个**（按步骤），平台差异用 `{{platform.*}}` 变量注入 |
| 视频成片 | 剪映草稿（人工进剪映导出） | **ffmpeg 自动合成 mp4**；剪映草稿降级为可选导出（继承 packages/jianying） |
| 平台差异 | 12 个流水线文件 | **1 个 `platforms.json` 参数表** |
| 网页端适配器（Playwright 驱动 ChatGPT 等） | 有 | **砍掉**（选择器易失效、维护贵，CLI/API 已够用） |
| Electron / MV / 图文流程 | 有 | 砍掉 / 延后 |

## 4. 数据模型（SQLite）

比 V1 大幅简化——没有 pipelines 多实例，一个选题对应一条产线：

```
topics       选题（id, title, source_type[text/image/footage], brief_json,
             status, created_at）
materials    素材（id, topic_id, kind[text/image/video], content/file_path）
steps        步骤实例（id, topic_id, step_id[analyze/title/script/storyboard/
             frames/tts/compose/review/adapt], status, provider_id,
             prompt_rendered, started_at, finished_at, error）
artifacts    产物（id, step_id, version, kind[text/image/audio/video/file],
             label, content/file_path, selected）
packages     发布包（id, topic_id, platform, video_path, title, caption,
             cover_paths_json, checklist_json, status[draft/exported/published]）
providers    引擎配置（id, kind[cli/api-text/api-image/api-video/tts],
             name, config_json, model, max_concurrency, enabled）
prompt_overrides  用户自定义模板（path, content, updated_at）
```

- 产物文件存 `workspace/<topic>/<step>/v<N>/`，重跑不覆盖旧版本（继承 V1）。
- `packages` 表是 V2 新概念：**发布包是一等公民**，记录"哪条母版派生到了哪个平台、发没发"。

## 5. 主线各步骤详细设计

### 5.0 analyze —— 素材理解（仅 A2 / B 模式，A1 跳过）

目的：把图片 / 实拍视频翻译成后续所有 AI 步骤都能引用的**结构化素材摘要**。

- **A2 参考图**：调 vision 引擎（api-text 多模态）读图 → 输出内容描述、可讲的点、风格关键词，注入 brief 供标题 / 口播稿 / 分镜引用；原图登记为出图参考与封面底图候选。
- **B 实拍视频**：
  1. ffprobe 探测时长 / 分辨率 / 横竖屏；
  2. ffmpeg 每 2 秒抽 1 帧（上限约 15 帧）；
  3. 抽帧交给 vision 引擎 → 输出素材内容摘要 + **可用片段标注**（每段 `start/end` 时间戳 + 该段画面里有什么、哪段最精彩）；
  4. 摘要与片段表注入后续步骤：口播稿围绕素材写，分镜表直接引用时间戳。
- 产物（素材摘要 JSON）入库为 artifact，**留可选人工卡点**——素材理解错了后面全错，允许人工修正摘要后再继续。

### 5.1 title —— 标题（5 候选）

- 与 V1 相同：一次生成 5 个候选按推荐度排序；`humanGate` 可选（全自动模式取第一）。
- **注意**：此处生成的是"母版工作标题"，不限平台字数——各平台的最终标题在 adapt 步骤按 `platform.titleMaxLen` 改写。

### 5.2 script —— 口播稿

- 输入：选中标题 + brief + 素材文本。
- 输出：口播稿全文 + 目标时长提示（默认 45~60 秒 ≈ 250~350 字，UI 可调）。

### 5.3 storyboard —— 分镜表（结构化 JSON）

这是成片链路的**数据枢纽**，输出严格 JSON：

```jsonc
{
  "scenes": [
    {
      "index": 1,
      "narration": "这一镜头要念的口播文本（≤40字，一口气能念完）",
      "subtitle": "屏幕字幕（通常=narration，可精简）",
      "source": "generated | footage",            // 画面来源（三种信息源在此统一）
      "visual": "画面描述（source=generated 时给出图API；A2 模式叠加参考图锚点）",
      "clip": { "start": 3.2, "end": 7.8 }         // source=footage 时：引用实拍素材的时间区间
    }
  ],
  "bgmMood": "轻快/悬疑/励志…"
}
```

- 镜头数 = 口播稿自然切分，通常 6~12 镜。
- 按信息源分配 `source`：A1 全部 `generated`；A2 全部 `generated`（带参考图锚点）；B 以 analyze 产出的片段表为骨架分配 `clip` 时间戳，AI 可补少量 `generated` 镜头（片头钩子 / 片尾收尾），也可全实拍。
- **不再包含 duration 字段**——每镜头时长由 TTS 音频实际时长驱动（见 5.5），这是 V1 分镜表没法自动成片的根本原因之一：让 AI 猜时长不可靠，让音频定时长才对。

### 5.4 frames —— 逐镜头声画物料

每个 scene 按 `source` 解析为三种画面物料之一，产物统一是"该镜头的视觉素材"：

| 来源 | 处理 | 说明 |
|---|---|---|
| `generated`（A1） | api-image 按 `visual` 出 1080×1920 竖图 | 原生比例出图，继承 V1 决策：不裁剪 |
| `generated`（A2） | 同上，叠加参考图锚点：引擎支持图生图则传参考图，否则把 vision 读图描述拼入提示词 | 保证整条片子与参考图内容 / 风格一致 |
| `footage`（B） | 不出图。登记素材文件 + `clip` 时间区间，合成时由 ffmpeg 切片 | 零生成成本 |

- 各镜头之间**并行**（受引擎 max_concurrency 限流）；单镜头可重抽 / 改选素材区间（继承 V1 的 reroll 机制）。
- 可选增强：api-video 图生视频，把静态图变成 3~5 秒动态片段（成本高，默认关）。

### 5.5 tts —— 逐镜头配音

- 每个 scene 的 `narration` 调 TTS 引擎，产出 `scene-N.mp3` + **实测时长（秒）**。
- 时长回填 scene：`scene.duration = 音频时长 + 0.3s 呼吸间隔`。
- 音色在 UI 全局选定（如晓晓/云希），语速可调（0.9x~1.2x）。
- 各镜头并行。

### 5.6 compose —— ffmpeg 合成母版成片 ★ 本次重构的核心

输入：frames 图片（或视频片段）+ tts 音频 + 分镜表。输出：`master.mp4`（1080×1920, H.264, yuv420p, 30fps, CRF 18, AAC）。

合成规则：

1. **每镜头按画面物料分三路渲染**，时长一律 = 该镜头 TTS 音频时长 + 0.3s（主时钟规则对三种信息源统一生效）：
   - 静态图 → `zoompan` Ken Burns 缓推（放大 1.0→1.08）；
   - 图生视频片段 → 直接裁到目标时长；
   - 实拍切片（B 模式）→ ffmpeg 从 `clip.start` 起截取目标时长，区间不够长时 0.9x 微慢放补齐；横屏素材居中 + 高斯模糊背景 pad 成 9:16；原声默认压至 -24dB 保留环境感（UI 可选：静音 / 保留原声不配旁白）。
2. **转场**：镜头间 `xfade`（0.3s 淡入淡出），朴素但稳。
3. **字幕**：分镜 `subtitle` + TTS 时长生成 ASS 文件，`subtitles` 滤镜烧录（底部 1/5 处，白字黑描边，思源黑体/微软雅黑）。
4. **音轨**：各镜头配音按顺序 concat；BGM（用户上传或内置几首免版权曲，按 `bgmMood` 选）压到 -18dB 混入，首尾淡入淡出。
5. **实现方式**：先把每镜头渲染成独立分段 mp4（可单独预览、单独重渲），再 concat 成母版——单镜头改画面/改配音只重渲一段，不用全片重来。

产出物同时保留：master.mp4、每镜头分段、ASS 字幕、（可选）剪映草稿目录（继承 packages/jianying，供想精修的人用——剪映从 V1 的主链路降级为 V2 的逃生通道）。

### 5.7 review —— 评审

继承 V1 双层机制：本地极限词/敏感词规则预检（继承 packages/review 词库）+ LLM 多维评分（钩子/清晰度/合规/平台适配）。评审对象是母版的标题+口播稿+封面；不过关自动按建议重生成一轮（继承 V1 全自动闭环）。

### 5.8 adapt —— 平台派生（V2 的第二个核心）

读 `platforms.json`，对勾选的每个平台产出一个**发布包**：

1. **视频转比例**：母版 9:16 → 目标比例。转换策略在参数表声明：
   - `9:16` → 直接复用母版（零成本）；
   - `3:4` → 中心裁切；
   - `16:9` / `1:1` → 母版居中 + 高斯模糊背景 pad（ffmpeg 一条命令）。
2. **标题/文案改写**：一次**便宜的小改写**（不是重新创作）——把母版标题+口播稿摘要交给文本引擎，按 `titleMaxLen`、`tagCount`、`voice` 产出该平台的标题 + 发布文案 + 话题标签。
3. **封面派生**：选中封面按平台尺寸用 sharp 缩放/裁切（继承 V1 的封面多尺寸派生，这正是 V1 里唯一做对了方向的地方）。
4. **注意事项清单**：参数表里的 `checklist` 直接进发布包。

发布包目录：

```
workspace/<topic>/packages/douyin/
  video.mp4         # 转好比例的成片
  title.txt         # 平台标题
  caption.txt       # 发布文案+标签
  cover.jpg         # 平台尺寸封面
  checklist.md      # 发布注意事项（发布时逐条勾选）
```

一键"导出全部发布包 ZIP"。发布本身仍然人工（继承 V1 不可动摇决策 ①：不做自动发布）。

## 10. 非目标（明确不做，比 V1 更狠）

1. **不做多平台独立流水线**——平台差异只活在 platforms.json。
2. **不做图文/长文/图集/MV 产线**——本项目只做短视频这一条线。
3. **不做自动发布**（继承 V1 决策，平台风控红线）。
4. **不做网页端 Playwright 适配器**。
5. **不做多用户/云端**，单机单用户，key 与数据不出本机。
6. **Electron 延后**：浏览器形态完全够用，全部里程碑完成后视需要再包壳。

## 11. 开发计划与里程碑

> 排序原则：**先打穿成片（M1~M3，用 A1 纯文本模式），再扩信息源（M4），再做派生（M5），最后锦上添花**。深度第一位，宽度只在深度打穿后扩。V1 是先铺宽度后补深度，V2 反过来。

| 阶段 | 内容 | 硬验收标准 |
|---|---|---|
| **M0 骨架** | monorepo、Fastify、node:sqlite 数据层（**source_type / scene.source 等信息源字段从第一天进 schema，避免返工**）、providers 接口（从 V1 迁移 cli/apiText/apiImage + mock）、WebSocket、最小前端 | 起服务、建选题、mock 出标题 |
| **M1 文本链路** | title(5候选+卡点) → script → storyboard(严格JSON+容错解析)，模板管理 | 真引擎跑出结构合法的分镜表 |
| **M2 声画物料** | frames 逐镜头出图（并行+单镜头重抽）、tts 引擎（Edge-TTS + tts-mock）逐镜头配音回填时长 | 每镜头有图有音有时长 |
| **M3 成片合成 ★** | ffmpeg 分段渲染（zoompan+ASS字幕）→ xfade concat → 混音 BGM → master.mp4；进度推送；单镜头重渲 | **不填任何 key，mock 全线跑通出一条能播放的演示 mp4**；真引擎出可发布成片（A1 模式深度打穿） |
| **M4 信息源扩展** | analyze 步骤（vision 读图 / ffmpeg 抽帧 + 片段标注）；B 实拍切片进 compose（截取/慢放/横屏模糊 pad/原声混音）；A2 参考图锚点出图 | **一段手机实拍 + 一句描述 → 全自动出带旁白字幕 BGM 的成片**；一张参考图 → 出内容风格一致的成片 |
| **M5 平台派生** | platforms.json、转比例（复用/裁切/模糊pad）、adapt-copy 改写、封面 sharp 派生、发布包目录 + 导出 ZIP | 勾 4 平台得到 4 个完整发布包，视频同源 |
| **M6 评审闭环 + 全自动** | 极限词预检（迁移V1词库）+ LLM 评分 + 不过自动重生成一轮；⚡全自动模式（选题进→发布包出） | 三种信息源均可一键全自动端到端无人值守跑完 |
| **M7 量产与打磨** | 选题批量队列（晚上跑早上挑）、失败自动重试1次、（可选）剪映草稿导出、（可选）Electron 壳 | 一次录 5 个选题排队全自动完成 |

每个里程碑配端到端冒烟脚本 + `pnpm typecheck` 作为合并门槛（继承 V1 实践）。

## 12. 关键风险与对策（继承 V1 踩坑 + 新增成片相关）

| 项 | 等级 | 对策 |
|---|---|---|
| **TTS 与字幕时间轴错位** | 高 | 铁律：时长永远以 TTS 音频实测时长为准，字幕/画面向音频对齐，绝不让 AI 估时长 |
| **ffmpeg Windows 字幕字体路径** | 中 | subtitles 滤镜路径需转义（`C\:/...`）；字体用 fontsdir 指定随项目分发的开源字体（思源黑体），不依赖系统字体 |
| **Edge-TTS 非官方接口可能失效** | 中 | tts 抽象为 provider kind，火山/Azure 即插即换；mock 兜底 |
| **zoompan 抖动/性能** | 低 | 输出前先把图 scale 到 2 倍再 zoompan 可消抖；分段渲染天然可并行 |
| 分镜 JSON 解析失败 | 中 | 继承 V1：严格 JSON 输出要求 + 容错提取（packages/shared/json）+ 失败展示原文人工修 |
| **素材理解偏差（B/A2 模式）** | 中 | analyze 摘要留可选人工卡点，改完摘要再跑后续；抽帧密度可调；vision 引擎可换 |
| 实拍素材过短 / 片段不够旁白念 | 中 | 分镜时把素材总时长告知 AI（旁白按素材体量写）；切片不足时 0.9x 慢放；仍不足提示用户精简口播稿 |
| SQLite `undefined` 绑定 | 已知 | 继承 V1 教训：写库前一律归一为 `null`/`""` |
| 出图中文渲染差 | 中 | 继承 V1：即梦为主；Grok 底图 + sharp 程序叠字兜底 |
| BGM 版权 | 中 | 只内置明确免版权曲目（或让用户自备），清单里标注来源 |
